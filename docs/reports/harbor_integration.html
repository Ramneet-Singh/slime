slime ↔ Harbor — Changes Needed to Support Agentic RL on Harbor-Style Tool Environments

Harbor (harbor-framework/harbor) is a Terminal-Bench-team framework for evaluating and optimizing agents in container-based tool environments and for generating rollouts for RL optimization. This report enumerates the concrete code-level changes needed inside this slime fork to consume Harbor-format tasks/agents/environments and drive multi-turn, tool-using RL training over them.

1. What Harbor specifies

From Harbor's AGENTS.md and README.md (commit on main), the framework is a Python 3.12 monorepo exposing a CLI (harbor, built on Typer) with these primitives:

ConceptDefinition (Harbor)What it gives RL training
Task Directory with task.toml (config: timeouts, resources, metadata), instruction.md (natural-language goal), environment/ (Dockerfile / env definition), tests/ (e.g. test.sh that writes a reward to /logs/verifier/reward.txt), optional solution/. Prompt source + verifiable reward + sandboxed execution surface.
Agent Subclass of BaseAgent with setup(), run(instruction, environment, context), and a SUPPORTS_ATIF flag. An "actor" wrapper; for RL we need a trainable agent that calls the slime-hosted policy.
Environment Subclass of BaseEnvironment (docker, daytona, e2b, modal, runloop, apple_container, gke, novita). The container/sandbox the agent runs against; tools = shell calls inside.
Trial / Job Trial = one agent × one task execution; Job = collection of trials. Maps to one "rollout" / one rollout batch in slime.
ATIF trajectory Agent Trajectory Interchange Format — Pydantic models under src/harbor/models/trajectories/. The canonical record of the multi-turn interaction (messages + tool calls + observations).
Verifier Reward written by tests/test.sh into /logs/verifier/reward.txt (or reward.json). Per-trajectory scalar reward → directly usable by GRPO/PPO.
Adapters Converters in adapters/{benchmark}/ that turn external benchmarks (SWE-Bench, Aider, BFCL, GAIA, …) into Harbor tasks. A free pipeline of 50+ tool-environment benchmarks for RL.
The CLI surface is harbor run --dataset <name@ver> --agent <agent> --model <model> --n-concurrent N --env <provider>. Harbor explicitly advertises "Generate rollouts for RL optimization" as a first-class feature, so the integration target is well-defined.

2. Gap analysis vs. slime today

slime already supports tool-based rollouts (examples/retool/, examples/search-r1/, examples/tau-bench/), but every tool environment is bespoke Python that runs in the rollout actor's process. Harbor adds three dimensions slime does not natively model:

Dimensionslime todayHarbor expectsGap
Environment lifecycle In-process Python (subprocess for ReTool sandbox) Container-backed environment with start/stop/build (docker / daytona / modal / e2b / gke / …) Major — need an environment manager that drives Harbor environments.
Task model JSONL/parquet prompts with a label string Directory of task.toml + instruction.md + tests; reward is an out-of-band script Medium — new data source + new reward path.
Agent ↔ model coupling Rollout fn calls SGLang router directly via HTTP BaseAgent is an opaque class; runs e.g. Claude Code, OpenHands inside the container, calling an LLM via LiteLLM Major — need a "trainable agent" that routes LLM calls back to the slime-hosted SGLang policy and records the trajectory in ATIF.
Reward Synchronous Python function in rm_hub/ or custom RM URL Verifier script writes /logs/verifier/reward.{txt,json} Medium — read-from-file adapter.
Trajectory format Sample with packed tokens + loss_mask ATIF Pydantic messages (role, content, tool calls) Medium — bidirectional converter, with care around tokenization & loss-mask boundaries.
Concurrency / placement Ray actors on GPU placement groups --n-concurrent across remote sandboxes (CPU-bound) Medium — need a CPU-only Ray pool (or asyncio pool) for environments, decoupled from GPU placement.
Multi-turn loss-mask Manual per-turn append (see retool) Implicit in ATIF message sequence Medium — generalise retool-style mask construction into a reusable helper.

3. Proposed integration design

Keep slime's core (Ray + Megatron + SGLang) untouched. Introduce a self-contained adapter package, modeled after the existing examples/retool/ + examples/tau-bench/ patterns but promoted into slime_plugins/harbor/ so that it is loadable via the standard --rollout-function-path / --custom-rm-path hooks.

┌─────────────────────────────────────────────────────────────┐ │ slime training loop │ │ (unchanged) │ └──────────────┬──────────────────────────────┬────────────────┘ │ rollout_manager.generate │ update_weights ▼ ▼ ┌──────────────────────────────┐ ┌────────────────────────────┐ │ slime_plugins/harbor │ │ SGLang engines (policy) │ │ ├─ rollout.generate_with_ │◀─▶│ (existing) │ │ │ harbor() │ └────────────────────────────┘ │ ├─ harbor_task_dataset.py │ │ ├─ harbor_agent.py │ ┌────────────────────────────┐ │ │ (TrainableHarborAgent) │──▶│ Harbor BaseEnvironment │ │ ├─ harbor_env_pool.py │ │ docker | daytona | modal │ │ ├─ atif_to_sample.py │ │ | e2b | gke | runloop ... │ │ ├─ harbor_reward.py │◀──│ reads /logs/verifier/... │ │ └─ arguments.py │ └────────────────────────────┘ └──────────────────────────────┘

Key design decisions

  1. Plugin, not core. Lives under slime_plugins/harbor/; mirrors how mbridge, megatron_bridge, rollout_buffer live today.
  2. Harbor SDK as a dependency. Add harbor (PyPI) to an opt-in extras section in pyproject.toml / requirements-harbor.txt; do not import it from core slime/.
  3. Reuse the existing rollout contract. The new function returns RolloutFnTrainOutput(samples=list[list[Sample]]) / RolloutFnEvalOutput(...) — no change to slime's training pipeline.
  4. Policy routing. Harbor agents speak LiteLLM. Provide a tiny LiteLLM-compatible adapter that points at the SGLang router (http://{sglang_router_ip}:{sglang_router_port}/v1) so any Harbor agent that uses the OpenAI-compatible interface (openhands-sdk, mini-swe-agent, aider, etc.) can be driven by the slime policy with no Harbor-side code change.
  5. Trajectory capture. Subclass BaseAgent with SUPPORTS_ATIF = True; consume the recorded ATIF trajectory at the end of the trial, then tokenize + build loss_mask.
  6. Two execution modes.
    • Mode A — "Harbor-driven": a CPU Ray pool runs Harbor trials; each trial logs prompts/completions; slime collects them after-the-fact.
    • Mode B — "slime-driven": the rollout function emulates the agent loop in process (à la generate_with_retool) but uses Harbor's environment as the tool sandbox (mount task, exec commands, run tests). Cleaner for token-level loss-mask control.
    We recommend implementing Mode B first (lower risk, full per-token control) and adding Mode A later for evaluating third-party agents.
  7. Async + concurrency. Use asyncio.TaskGroup over Harbor environments (Harbor itself is asyncio-native), gated by a --harbor-concurrency semaphore. The existing async-rollout machinery (slime/rollout/fully_async_rollout.py + train_async.py) is the right substrate.

5. New files / packages (with responsibilities)

Aslime_plugins/harbor/harbor_task_dataset.py

Wraps Harbor's task discovery (CLI: harbor datasets list; SDK: harbor.dataset / harbor.registry) and exposes an iterator of seed Sample objects whose prompt is instruction.md, whose label stores the task path / id, and whose metadata carries the task descriptor (timeouts, environment image, expected resources). This object is the value plugged into --prompt-data via a small custom data-source loader.

Bslime_plugins/harbor/harbor_env_pool.py

Owns a pool of Harbor environments per concurrent rollout slot:

  • acquire(task) → env — build/start container (Docker / Daytona / Modal / …); supports prebuilt cache.
  • exec(env, cmd) → stdout/stderr/exit_code — generic tool surface.
  • run_verifier(env) → reward — runs tests/test.sh, reads /logs/verifier/reward.{txt,json}.
  • release(env, delete=True) — stop and reclaim.

Backed by asyncio.Semaphore; respects --harbor-concurrency and --harbor-env-provider.

Cslime_plugins/harbor/litellm_to_sglang.py

Thin OpenAI-compatible HTTP server (or an in-process LiteLLM custom provider) that exposes POST /v1/chat/completions backed by the slime SGLang router. Run as a sidecar in each rollout worker. This makes off-the-shelf Harbor agents (openhands-sdk, aider, mini-swe-agent, …) talk to the trainable policy without modification — they already speak LiteLLM.

Dslime_plugins/harbor/harbor_agent.py

class TrainableHarborAgent(BaseAgent) with SUPPORTS_ATIF = True. Its run() drives a multi-turn loop: read instruction → call the SGLang policy directly (capturing token ids and log-probs) → parse tool calls → exec via the Harbor environment → append observation → repeat. Crucially, it records both (i) the canonical ATIF messages (for Harbor) and (ii) the slime-side per-turn token ids and assistant/tool-token boundaries needed for the loss mask.

Eslime_plugins/harbor/atif_to_sample.py

Pure function: atif_trajectory + tokenizer + reward → Sample. Mirrors the loss-mask construction in examples/retool/generate_with_retool.py (assistant tokens → mask 1; system / user / tool-observation tokens → mask 0). Stores the original ATIF blob in sample.metadata["atif"] for downstream debugging.

Fslime_plugins/harbor/harbor_reward.py

Tries /logs/verifier/reward.json first, then reward.txt; supports scalar, dict ({"reward": 1.0, "breakdown": {...}}), and optionally accepts a user-supplied shaper (--harbor-reward-shaper-path). Returns a float or a dict that slime PPO/GRPO already understands (Sample.reward is typed float | dict | None).

Gslime_plugins/harbor/rollout.py

The entry point: async def generate(args, sample: Sample, sampling_params) -> Sample and a batch-level def generate_rollout(args, rollout_id, data_buffer, evaluation: bool=False) -> RolloutFnTrainOutput. Performs: acquire env → invoke TrainableHarborAgent → atif_to_sample → run_verifier → release env → return sample(s) grouped for GRPO (one task ⇒ N rollouts grouped by group_index).

6. Modified files (surgical changes)

6.1 slime/utils/data.py

Today, Dataset reads jsonl / parquet files. Add a thin extension point: if args.prompt_data begins with harbor://, defer to the harbor task dataset iterator. Implementation is ≈20 lines and avoids importing Harbor unless the prefix is used.

6.2 slime/utils/arguments.py

No new code paths needed — slime already calls add_custom_arguments(parser). The new harbor flags live in slime_plugins/harbor/arguments.py and are wired in at launch via:

--custom-arguments-path slime_plugins.harbor.arguments:register

6.3 slime/rollout/rm_hub/__init__.py

Ensure async_rm tolerates a None reward (the Harbor rollout fn fills sample.reward itself). Most likely already the case — just add a guard + test.

6.4 slime/ray/placement_group.py

Add an optional "harbor" placement group with num_cpus >= N, num_gpus = 0 when args.harbor_num_cpu_workers > 0. Used by harbor_env_pool.py to schedule env containers on nodes that have spare CPU but no policy GPU pressure.

6.5 slime/utils/types.py

Optional: add tool_calls: list | None = None to Sample for first-class structured tool-call storage. If we'd rather not touch core types, store under metadata["tool_calls"].

9. The new rollout function (sketch)

Modeled on examples/retool/generate_with_retool.py:

slime_plugins/harbor/rollout.py (proposed)
async def _one_trial(args, task_sample, sampling_params, env_pool, sglang_url, tokenizer):
    async with env_pool.acquire(task_sample.metadata["task"]) as env:
        await env.start(force_build=False)

        agent = TrainableHarborAgent(
            sglang_url=sglang_url,
            tokenizer=tokenizer,
            max_turns=args.harbor_max_turns,
            sampling_params=sampling_params,
            tool_allowlist=args.harbor_tool_allowlist,
        )
        await agent.setup(env)
        await agent.run(
            instruction=task_sample.prompt,
            environment=env,
            context=AgentContext(...),
        )

        raw_reward = await env_pool.run_verifier(env)
        reward     = harbor_reward.shape(raw_reward, args)

        sample = atif_to_sample(
            atif=agent.trajectory,
            tokenizer=tokenizer,
            reward=reward,
            seed=task_sample,
        )
        return sample

async def generate_rollout(args, rollout_id, data_buffer, *, evaluation=False):
    sglang_url = f"http://{args.sglang_router_ip}:{args.sglang_router_port}/generate"
    env_pool   = HarborEnvPool(args)            # built once, reused across calls
    tokenizer  = load_tokenizer(args)
    sem = asyncio.Semaphore(args.harbor_concurrency)

    async def run_group(task_sample):
        async with sem:
            # GRPO: n rollouts per task, share the same env image (warm cache)
            samples = await asyncio.gather(*[
                _one_trial(args, task_sample, sampling_params, env_pool, sglang_url, tokenizer)
                for _ in range(args.harbor_rollouts_per_task)
            ])
            for i, s in enumerate(samples):
                s.group_index = task_sample.index
                s.index       = i
            return samples

    task_batch = data_buffer.next_batch(args.rollout_batch_size)
    grouped    = await asyncio.gather(*[run_group(t) for t in task_batch])

    if evaluation:
        return RolloutFnEvalOutput(data={t.label: {"reward": np.mean([s.reward for s in g])}
                                          for t, g in zip(task_batch, grouped)})
    return RolloutFnTrainOutput(samples=grouped, metrics=_harbor_metrics(grouped))

10. Rewards — Harbor verifier → slime reward

Harbor's contract: the verifier writes a reward file inside the container after the agent finishes. Concretely /logs/verifier/reward.txt (a single number) or /logs/verifier/reward.json (object with at least a reward key and optional breakdown). Our reader:

async def read_reward(env) -> float | dict:
    try:
        raw = await env.read_file("/logs/verifier/reward.json")
        obj = json.loads(raw)
        return obj if isinstance(obj, dict) else float(obj)
    except FileNotFoundError:
        raw = await env.read_file("/logs/verifier/reward.txt")
        return float(raw.strip())
    except Exception as e:
        logger.warning("Harbor verifier reward unavailable: %s", e)
        return 0.0

The returned value flows into Sample.reward directly (slime PPO/GRPO already supports both float and dict rewards — see slime/utils/types.py).

11. ATIF trajectory ↔ slime Sample mapping

The key invariant for RL is: only tokens that the trainable policy emitted contribute to the loss. ATIF makes this explicit because each message carries its role. The conversion rule is:

ATIF message roleTokenized contributionloss_mask
systemchat template wrapper0
userincluded verbatim0
assistant (text)included with chat-template wrapper1
assistant.tool_calls (function name + JSON args)included as tool-call special tokens1
tool / tool_resultincluded (environment observation)0

Because the existing TrainableHarborAgent generates assistant tokens by streaming from the SGLang router, it is more robust to build tokens and loss_mask incrementally during generation (as ReTool does) rather than re-tokenizing the ATIF post-hoc, which can introduce subtle whitespace/BPE drift. atif_to_sample therefore primarily provides a fallback path for Mode A (third-party agents).

Gotcha: slime's training-side max length is context_parallel_size * max_tokens_per_gpu; the agent loop must clamp each turn's max_new_tokens to the remaining budget exactly as generate_with_retool.py does — otherwise the partition/batch code can assert or OOM.

12. Async / scaling notes

  • Harbor environments are container-bound — start-up can dominate. Use --harbor-cache-built-envs + pre-build per task image at rollout_manager init time (a new RolloutManager.warmup() hook may help).
  • For large datasets (Terminal-Bench-2.0 has hundreds of tasks), pair with train_async.py + fully_async_rollout.py so training can overlap with the long tool-execution tail (similar to APRIL's partial-rollout idea).
  • Per-trial timeouts are critical: enforce asyncio.wait_for at the trial level and return a Sample.Status.ABORTED with reward = 0 rather than blocking the rollout step.
  • Network egress: if the agent uses MCP / web tools, route through Harbor's --ae env-var injection and never bake API keys into the image.
  • For multi-GPU policy serving, the SGLang router already load-balances; no Harbor-side change is needed.

13. Caveats & open questions

  • Tokenization fidelity for off-the-shelf agents (Mode A). When Claude-Code / OpenHands generates text through their own client, we only see the final ATIF. Round-tripping through the trainable tokenizer can mis-align tokens with the original sampler's log-probs. Mode B avoids this entirely; Mode A should constrain the loss to assistant character spans via re-tokenization + a tolerance check.
  • Tool-call special tokens. Some chat templates (Qwen, GLM) tokenize tool calls into model-specific special tokens; we must use the model's apply_chat_template(tools=...) rather than a hand-rolled Jinja template like ReTool currently does, to preserve compatibility with the policy.
  • Dataset registry licensing. Harbor's registry.json is large (≈13 MB). We should not vendor it; instead, depend on harbor at runtime and pull from its registry on demand.
  • Reward sparsity. Many Harbor tasks are pass/fail — combine with shaping (--harbor-reward-shaper-path) such as turn-count or partial-test credit, or use GRPO with large group size to amortise variance.
  • Container resource governance. When using docker provider co-located with GPUs, set explicit cpus/memory limits in task.toml and via env-pool defaults to avoid starving Megatron actors.
  • Evaluation. Reuse the same rollout fn with evaluation=True path and the --eval-config machinery to evaluate on held-out Harbor splits with stricter timeouts.
