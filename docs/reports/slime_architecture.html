1. Overview

slime is an SGLang-native, Megatron-backed RL post-training framework. It is the RL stack behind GLM-4.5/4.6/4.7/5/5.1 and supports Qwen, DeepSeek-V3 and Llama-3 families. Its design splits cleanly into three logical modules wired together by Ray:

Training

Megatron-LM actor (+ optional critic), parameter-synced after each step.

slime/backends/megatron_utils

Rollout

SGLang engines behind sglang-router; produce trajectories & rewards.

slime/rollout + slime/backends/sglang_utils

Data Buffer

Bridge that owns prompts, custom data, generation hooks and feeds training.

slime/ray/rollout.py (RolloutManager)
The framework is intentionally plug-pointed at the rollout function — agentic, multi-turn, tool-calling, search, MCP, etc. are all implemented as custom --rollout-function-path modules without touching the training backend.

3. High-Level Architecture

The three modules communicate through Ray object references. The RolloutManager is the central scheduler that owns the SGLang engines and the data buffer, and exposes async-style methods (generate, eval, onload_weights, …) called by the training driver.

4. Entry Points

4.1 train.py — synchronous loop

/train.py
def train(args):
    configure_logger()
    pgs = create_placement_groups(args)                                 # 1. GPU allocation
    init_tracking(args)

    rollout_manager, num_rollout_per_epoch = create_rollout_manager(    # 2. Bring up SGLang
        args, pgs["rollout"])

    router_addr = ray.get(rollout_manager.get_metrics_router_addr.remote())
    update_tracking_open_metrics(args, router_addr)

    actor_model, critic_model = create_training_models(                  # 3. Build Megatron actors
        args, pgs, rollout_manager)

    actor_model.update_weights()                                         # 4. Push weights to rollout

    for rollout_id in range(args.start_rollout_id, args.num_rollout):    # 5. Train loop
        rollout_data_ref = ray.get(rollout_manager.generate.remote(rollout_id))
        ray.get(actor_model.async_train(rollout_id, rollout_data_ref))
        ...
        actor_model.update_weights()                                     # sync each step
        if should_run_periodic_action(rollout_id, args.eval_interval, ...):
            ray.get(rollout_manager.eval.remote(rollout_id))

The loop is deliberately small: generate → train → sync weights → (optional) eval. Critic, offloading, and save are interleaved via flags (--use-critic, --offload-rollout, --offload-train, --save-interval).

4.2 train_async.py — fully async loop

Used by the fully async example (examples/fully_async/). Rollout and train run concurrently with a configurable staleness budget; weight sync is overlapped with the next generation step (similar in spirit to Relax/DCS).

Rollout output contract

slime/rollout/base_types.py
@dataclass
class RolloutFnTrainOutput:
    samples: list[list[Sample]]      # list of groups (for GRPO etc.)
    metrics: dict[str, Any] = None

@dataclass
class RolloutFnEvalOutput:
    data: dict[str, dict[str, Any]]
    metrics: dict[str, Any] = None

def call_rollout_fn(fn, *args, evaluation: bool, **kwargs):
    output = fn(*args, **kwargs, evaluation=evaluation)
    if not isinstance(output, (RolloutFnTrainOutput, RolloutFnEvalOutput)):
        output = RolloutFnEvalOutput(data=output) if evaluation \
                 else RolloutFnTrainOutput(samples=output)
    return output

All custom rollout functions (search-R1, ReTool, τ-bench, MCP, etc.) build Sample objects and return them via the output dataclasses above. loss_mask is critical for tool-based training: only model-generated tokens are scored.

9. slime_plugins/

  • mbridge / megatron_bridge — model-architecture bridges (HF → Megatron parameter mapping for Qwen, DeepSeek, GLM, Llama).
  • models — opt-in model definitions, MoE helpers, custom layers.
  • rollout_buffer — alternate buffer implementations (e.g. for asynchronous & partial-rollout schemes).

Plugins are not imported by default — they are loaded by load_function when a CLI flag (e.g. --rollout-function-path) names them. Keeping them out-of-core lets external projects (Relax, APRIL, qqr, TritonForge…) ship rollouts without forking slime.

                   "return_logprob": True}

        # parse <code>…</code> or final answer; if code:
        async with SEMAPHORE:
            result = await tool_registry.execute_tool("code_interpreter", {"code": code})

This is the pattern that any future tool-environment integration (Harbor, MCP, etc.) needs to follow: assemble multi-turn tokens, set loss_mask=0 on environment-generated tokens, return a Sample with reward.

11. End-to-End Data Flow

1. Launch train.py / train_async.py └─ parse_args() (slime/utils/arguments.py) └─ create_placement_groups(args) (Ray pgs for actor / rollout / critic) └─ create_rollout_manager(args, pg) → RolloutManager Ray actor └─ ServerGroup.start_engines() → N × SGLangEngine + sglang_router 2. Per rollout step (rollout_id = 0 … num_rollout-1) a. rollout_manager.generate.remote(rollout_id) ├─ Dataset → next batch of prompts ├─ call_rollout_fn(args.rollout_function_path or default sglang_rollout) │ ├─ posts to router /generate (single- or multi-turn / tool-call) │ ├─ may call external tools (search, sandbox, MCP, …) │ └─ returns RolloutFnTrainOutput(samples=[[Sample, …], …]) ├─ rm_hub.async_rm(...) → fills Sample.reward ├─ filter_hub (optional dynamic filter / dedup / repetition) └─ returns ObjectRef to packed batch tensors b. actor_model.async_train(rollout_id, rollout_data_ref) ├─ Megatron forward / loss (PPO / GRPO / DPO / OPD) ├─ optimizer step (+ optional critic step) └─ optional offload_train() c. actor_model.update_weights() ├─ Megatron → SGLangEngine via NCCL broadcast (or disk) └─ optional onload/offload of KV cache & weights d. (periodic) save() → checkpoint, eval() → rollout_manager.eval.remote() 3. Teardown rollout_manager.dispose.remote() finish_tracking(args)
